# File: dispatcher/src/test/java/com/cloudbalancer/dispatcher/integration/SecurityIntegrationTest.java

## Overview

`SecurityIntegrationTest` is a critical integration test suite for the `dispatcher` module, responsible for validating the end-to-end security lifecycle of the application. It ensures that authentication, authorization, and token management (login, access, and refresh) function correctly within a real Spring Boot environment.

**Note:** This file is a **HOTSPOT**. It is in the top 25% for both change frequency and complexity. Modifications to this file carry a high risk of introducing regressions in the authentication flow, which is fundamental to the system's security.

## Public API

The class provides the following test methods to verify the security contract:

*   `setUp()`: Initializes the `RestClient` with the dynamically assigned random port before each test execution.
*   `fullAuthFlow_loginAccessRefreshAccess()`: Executes a complete authentication lifecycle:
    1.  Authenticates via `/api/auth/login`.
    2.  Validates the presence of `accessToken` and `refreshToken`.
    3.  Accesses a protected resource (`/api/tasks`) using the `accessToken`.
    4.  Refreshes the session via `/api/auth/refresh`.
    5.  Verifies that the new `accessToken` grants access to protected resources.
*   `unauthenticatedRequestReturns401()`: Verifies that requests to protected endpoints without valid credentials correctly trigger a `401 Unauthorized` error.

## Dependencies

## Usage Notes

### Testing Lifecycle
This test assumes that the application context is seeded with an initial admin user (typically via a `DataSeeder` component). If the data seeding logic changes, these tests may fail due to missing credentials.

### Integration with Controllers
These tests directly interact with the `AgentController` (or related task-management controllers) via the `/api/tasks` endpoint. Any changes to the security configuration or the endpoint pathing in the `dispatcher` module will require corresponding updates to these tests.

### Example: Adding a New Security Test
To add a new test case for a different role or permission, follow this pattern:

```java
@Test
void testNewRoleAccess() {
    // 1. Obtain token for specific role
    // 2. Perform request
    // 3. Assert status code or response body
}
```

### Potential Pitfalls
*   **Port Conflicts**: Since it uses `RANDOM_PORT`, ensure that the environment allows for ephemeral port allocation.
*   **Token Expiry**: The `fullAuthFlow` test relies on the validity of the tokens generated by the auth service. If the token TTL (Time-To-Live) is configured to be extremely short, tests might flake.
*   **Database State**: Because this is an integration test, it relies on the state of the database. Ensure `TestContainersConfig` is correctly cleaning up state between test runs to prevent cross-test pollution.
