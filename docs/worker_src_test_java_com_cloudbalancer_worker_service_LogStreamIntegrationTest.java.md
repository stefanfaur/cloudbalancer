# File: worker/src/test/java/com/cloudbalancer/worker/service/LogStreamIntegrationTest.java

## Overview

`LogStreamIntegrationTest` is a critical integration test suite within the `worker` module. It validates the end-to-end log capturing and serialization pipeline, ensuring that log output generated by task executors is correctly captured, timestamped, and serialized into the `LogMessage` JSON format. This format is identical to the schema required for publication to the `tasks.logs` Kafka topic.

**Note:** This file is identified as a **HOTSPOT** (top 25% for change frequency and complexity). It is a high-risk area for bugs, as changes to the log serialization logic or the `LogCallback` interface can directly impact the observability of task execution in the production environment.

## Public API

The class contains the following test methods:

*   `logCallbackProducesValidLogMessageJson()`: Verifies that the `LogCallback` correctly maps inputs to a `LogMessage` object and that the resulting JSON is valid and supports round-trip deserialization.
*   `pythonExecutorStreamsLogsThroughCallback()`: An integration test that invokes the `PythonExecutor` to run a script, confirming that stdout and stderr streams are correctly intercepted by the `LogCallback` and preserved with appropriate metadata.
*   `logMessagesPreserveSpecialCharacters()`: Ensures that Unicode characters, file paths, and JSON-reserved characters (like quotes and backslashes) are correctly handled during serialization and deserialization.
*   `emptyLogLineSerializesCorrectly()`: Validates that empty strings in log lines do not cause serialization errors or data loss.

## Usage Notes

### Implementation Rationale
The tests are designed to run in isolation without requiring a Spring context, Kafka brokers, or other external infrastructure. This allows for rapid feedback loops during development. By using `CopyOnWriteArrayList` and manual callback invocation, the tests simulate asynchronous log streaming behavior reliably.

### Potential Pitfalls
*   **Serialization Mismatches**: Since this test validates the contract for the `tasks.logs` Kafka topic, any change to the `LogMessage` class structure must be reflected here. Failure to update these tests when modifying `LogMessage` will lead to downstream failures in log consumers.
*   **Timestamp Precision**: Tests involving `Instant` should be handled carefully; while `JsonUtil` generally handles ISO-8601, ensure that the serialization configuration remains consistent across the `worker` and `dispatcher` modules.
*   **Hotspot Caution**: As a high-activity file, ensure that any modifications to the `LogCallback` interface are accompanied by corresponding updates to these integration tests to prevent breaking the log pipeline.
