# Module: .claude

## Overview

The `.claude` module serves as the foundational codebase for **CloudBalancer**, a distributed task execution and orchestration platform. The system is architected as a modular, event-driven ecosystem designed to manage the lifecycle of computational tasks across a cluster of worker nodes.

The module is organized into four primary functional areas:
*   **Common**: Contains shared domain models, event definitions, and executor abstractions used by all components.
*   **Dispatcher**: The control plane responsible for task scheduling, worker registry management, artifact storage, and cluster monitoring.
*   **Worker**: The data plane responsible for executing tasks, reporting telemetry, and managing local execution environments (Docker, Python, Shell).
*   **Metrics-Aggregator**: A dedicated service for ingesting, persisting, and querying cluster-wide performance and health telemetry.

## Public API Summary

### Core Domain Models
*   **Task Management**: `TaskEnvelope`, `TaskDescriptor`, `TaskState`, and `TaskResult` provide the state machine and data structures for tracking task lifecycles.
*   **Worker Management**: `WorkerInfo`, `WorkerCapabilities`, and `WorkerHealthState` define the operational profile and status of cluster nodes.
*   **Events**: A sealed hierarchy (`CloudBalancerEvent`) including `TaskSubmittedEvent`, `TaskAssignedEvent`, `TaskCompletedEvent`, and `WorkerHeartbeatEvent` facilitates asynchronous communication via Kafka.

### Execution Abstractions
*   **TaskExecutor**: The primary interface for task execution. Implementations include:
    *   `DockerExecutor`: Runs tasks in isolated containers.
    *   `PythonExecutor`: Manages Python script execution in virtual environments.
    *   `ShellExecutor`: Executes arbitrary shell commands with security constraints.
    *   `SimulatedExecutor`: Used for testing and resilience simulation.

### Dispatcher API
*   **TaskController**: REST endpoints for submitting and monitoring tasks.
*   **AdminController**: Administrative interface for switching scheduling strategies (e.g., `RoundRobin`, `LeastConnections`, `ResourceFit`).
*   **ArtifactController**: Manages the storage and retrieval of task-related input/output artifacts.

### Metrics & Monitoring
*   **MetricsController**: Exposes cluster-wide performance metrics and historical worker data.
*   **HeartbeatTracker**: Monitors worker liveness and triggers recovery workflows upon node failure.

## Architecture Notes

*   **Event-Driven Communication**: The system relies heavily on Apache Kafka for inter-service communication. All state changes (task lifecycle, worker registration, metrics) are propagated as events, ensuring loose coupling between the Dispatcher and Worker nodes.
*   **Scheduling Pipeline**: The Dispatcher utilizes a pluggable scheduling pipeline. It employs a sequence of `WorkerFilter` (e.g., `HealthFilter`, `ResourceSufficiencyFilter`) and `WorkerScorer` (e.g., `QueueDepthScorer`) components to determine the optimal node for task assignment.
*   **Persistence Layer**: The system uses JPA with PostgreSQL for state persistence. Specific converters (e.g., `TaskDescriptorConverter`, `ExecutionHistoryConverter`) are implemented to handle the mapping of complex domain objects to JSONB columns, ensuring flexibility in task metadata storage.
*   **Security**: The Dispatcher and Metrics-Aggregator enforce security via JWT-based authentication and authorization. A `RateLimitFilter` (using `bucket4j`) is implemented to protect API endpoints from abuse.
*   **Resilience**: Fault tolerance is managed through:
    *   **Circuit Breakers**: Implemented via Resilience4j to protect against cascading failures in Kafka producers and service calls.
    *   **Chaos Monkey**: A dedicated service and controller allow for the injection of latency and worker failures to verify system resilience.
    *   **Retry Logic**: The `RetryScanner` service periodically reconciles failed tasks, applying backoff strategies to manage retries.
